# /plan 命令

## 概述

`/plan` 命令用于在编写任何代码之前创建详细的实现计划。这是 ECC 工作流的第一步，也是最重要的一步。

## 用途

- 创建新功能的实现计划
- 制定重构策略
- 分析复杂任务的实现步骤
- 识别依赖关系和风险
- 明确需求和实现路径

## 何时使用

使用 `/plan` 命令的场景：

1. **开始新功能开发**
   - 添加新的 API 端点
   - 实现新的业务逻辑
   - 集成新的服务

2. **进行重大架构变更**
   - 修改数据库架构
   - 改变应用架构
   - 重构核心模块

3. **复杂重构工作**
   - 提取公共逻辑
   - 优化性能
   - 改进代码结构

4. **需求不明确时**
   - 需要明确实现步骤
   - 需要评估实现难度
   - 需要识别潜在问题

## 工作原理

### 规划流程

```
用户执行 /plan 命令
    ↓
[1] 分析需求并重述
    ↓
[2] 识别技术方案
    - 选择技术栈
    - 设计架构
    - 定义接口
    ↓
[3] 分解实现步骤
    - 分阶段实现
    - 定义每个阶段任务
    - 识别依赖关系
    ↓
[4] 评估风险和复杂度
    - 识别技术风险
    - 评估实现难度
    - 提出缓解措施
    ↓
[5] 创建计划文档
    - 需求重述
    - 实现步骤
    - 风险分析
    - 预估时间
   ↓
[6] 等待用户确认
   - 用户确认后开始实施
   - 用户可以要求修改
```

### 计划文档结构

规划代理生成的计划文档通常包含以下部分：

```markdown
# 实现计划：[功能名称]

## 需求重述
- 明确的功能描述
- 功能目标
- 成功标准

## 技术方案
- 技术栈选择
- 架构设计
- 接口定义
- 数据模型

## 实现步骤
### 阶段 1：[阶段名称]
- 步骤 1
- 步骤 2

### 阶段 2：[阶段名称]
- ...

## 依赖关系
- 前置条件
- 依赖模块
- 外部服务

## 风险分析
- 技术风险
- 业务风险
- 缓解措施

## 预估复杂度
- 实现难度
- 预估时间
```

## 使用示例

### 示例 1: 规划新功能

```bash
用户: /plan 我需要添加用户认证功能，支持邮箱密码登录和 JWT Token

ECC (planner 代理):
# 实现计划：用户认证功能

## 需求重述
实现用户认证系统，包括：
- 用户注册（邮箱、密码）
- 用户登录（邮箱、密码）
- JWT Token 生成和验证
- 密码加密存储

## 技术方案
- 使用 Spring Security
- 使用 BCrypt 加密密码
- 使用 JJWT 生成 JWT Token
- JWT 有效期：7 天

## 实现步骤
### 阶段 1：数据库模型
1. 创建 User 实体（id, email, password, createdAt）
2. 创建 UserRepository
3. 添加索引到 email 字段

### 阶段 2：认证服务
1. 创建 AuthService
2. 实现密码加密和验证
3. 实现 JWT Token 生成和验证
4. 实现用户注册逻辑

### 阶段 3：API 端点
1. 创建 AuthController
2. 实现 POST /api/auth/register
3. 实现 POST /api/auth/login
4. 添加输入验证

### 阶段 4：安全配置
1. 配置 Spring Security
2. 实现 JWT 过滤器
3. 配置受保护的端点

## 风险分析
- 密码加密：使用 BCrypt
- JWT 安全：使用强密钥
- 输入验证：防止注入攻击

## 预估复杂度：中等
- 实现时间：4-6 小时

等待确认：proceed?
```

### 示例 2: 规划重构

```bash
用户: /plan 重构订单服务，提取支付逻辑到独立的服务

ECC (planner 代理):
# 实现计划：提取支付服务

## 需求重述
将订单服务中的支付逻辑提取到独立的支付服务，使系统更模块化。

## 技术方案
- 创建新的 Payment Service 模块
- 使用 Feign Client 进行服务间调用
- 保持 API 兼容性

## 实现步骤
### 阶段 1：创建支付服务
1. 创建 PaymentService 项目
2. 实现支付核心逻辑
3. 实现 Feign Client 接口

### 阶段 2：迁移订单服务
1. 更新订单服务调用支付服务
2. 移除订单服务中的支付逻辑
3. 更新测试

### 阶段 3：验证和测试
1. 运行集成测试
2. 验证功能完整性
3. 性能测试

## 风险分析
- 服务间通信延迟
- 数据一致性
- 回滚策略

## 预估复杂度：高
- 实现时间：8-12 小时

等待确认：proceed?
```

## 计划确认

规划完成后，ECC 会等待你的确认：

### 确认选项

- **"yes" 或 "proceed"** - 开始实施计划
- **"no"** - 取消计划
- **"modify: [修改建议]"** - 修改计划
- **"skip phase X"** - 跳过某个阶段
- **"change order: [新的顺序]"** - 调整步骤顺序

### 修改示例

```bash
用户: modify: 增加单元测试阶段，在阶段 4 之前添加

ECC: 更新计划... [重新生成计划]

用户: yes  # 确认修改后的计划

ECC: 开始实施计划...
```

## 最佳实践

### 1. 总是从 /plan 开始

**原因**:
- 明确需求和实现步骤
- 识别潜在风险和依赖
- 提高开发效率，减少返工

**示例**:
```bash
# 好的做法
用户: /plan 添加用户认证功能
[获得详细计划]
用户: yes
[开始实施]

# 不好的做法
用户: 直接开始写代码，没有规划
[可能导致返工和遗漏]
```

### 2. 详细描述需求

**原因**:
- 确保计划准确反映需求
- 避免理解偏差
- 提高计划质量

**示例**:
```bash
# 好的做法
用户: /plan 添加用户认证功能，支持邮箱密码登录、JWT Token、刷新 Token

# 不好的做法
用户: /plan 添加用户认证功能
```

### 3. 审查计划文档

**原因**:
- 确保计划完整
- 识别遗漏的步骤
- 评估计划的合理性

**检查清单**:
- [ ] 需求重述准确
- [ ] 技术方案合理
- [ ] 实现步骤完整
- [ ] 风险分析充分
- [ ] 时间估算合理

### 4. 不要害怕修改计划

**原因**:
- 计划不是一成不变的
- 根据实际情况调整
- 提高最终实现质量

**示例**:
```bash
用户: /plan 添加用户认证功能
[获得计划]
用户: modify: 增加邮箱验证功能
[更新计划]
用户: yes
```

## 常见问题

**Q: 简单的功能也需要 /plan 吗？**

A: 建议总是使用 `/plan`，即使简单的功能。这有助于培养良好习惯，并确保不遗漏任何步骤。

**Q: 计划太长怎么办？**

A: 你可以要求简化计划，或者修改计划聚焦核心步骤。规划代理会根据你的反馈调整。

**Q: 计划可以修改吗？**

A: 可以。你可以使用 "modify" 选项要求修改计划，或者在接受后在实际开发中调整。

**Q: 规划需要多长时间？**

A: 通常 1-5 分钟，取决于需求的复杂度。

**Q: 可以跳过规划直接开发吗？**

A: 技术上可以，但不推荐。规划是 ECC 工作流的重要部分，跳过规划可能导致效率低下和质量问题。
