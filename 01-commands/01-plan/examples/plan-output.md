# 实现计划：电商平台限时秒杀功能

## 需求重述

为电商平台实现一个完整的限时秒杀系统，包含以下核心功能：

1. **活动管理**: 创建、更新、查询、删除秒杀活动
2. **用户抢购**: 用户在活动时间窗口内抢购商品
3. **并发控制**: 保证库存扣减的原子性，防止超卖
4. **结果通知**: 实时通知用户抢购结果
5. **自动调度**: 定时启用和关闭秒杀活动

**业务约束**:
- 单用户每个活动限购 1 件
- 先到先得，库存售完即止
- 活动仅在指定时间窗口内有效
- 支持高并发抢购（目标 1000 QPS）

## 技术方案

### 并发控制方案

**方案对比**:

| 方案 | 优点 | 缺点 | 评分 |
|------|------|------|------|
| 数据库乐观锁 | 实现简单，事务保证 | 高并发时性能差，数据库压力大 | ⭐⭐ |
| Redis + Lua 脚本 | 高性能，原子性保证，减少数据库压力 | 需要维护缓存一致性 | ⭐⭐⭐⭐ |
| 分布式锁 | 灵活，支持复杂逻辑 | 性能开销大，实现复杂 | ⭐⭐⭐ |

**推荐方案**: **Redis + Lua 脚本**

**理由**:
- 高并发场景下性能最优
- Lua 脚本保证原子性，无需额外锁
- 减少数据库压力，提高系统整体性能
- 实现复杂度适中

### 缓存策略

**缓存对象**:
- 秒杀活动信息（预热到 Redis）
- 商品库存（Redis 计数器）
- 用户抢购记录（Redis Set，防止重复）

**缓存更新策略**:
- 活动信息：写时更新 + 定期同步
- 库存：仅通过 Lua 脚本更新
- 用户记录：抢购成功后写入，活动结束后清理

**缓存预热**:
- 活动开始前 5 分钟预热到 Redis
- 包括活动信息、库存初始值

### 消息通知方案

使用 RabbitMQ 实现异步通知：

- **交换机**: flash-sale-events
- **队列**: flash-sale-notifications
- **路由键**:
  - `order.success`: 抢购成功通知
  - `order.failed`: 抢购失败通知

## 实现步骤

### 阶段 1：数据库设计（约 1-2 小时）

**任务列表**:

1. 创建数据库表
   - `flash_sale_activity`: 秒杀活动表
     - id (PK)
     - product_id (商品ID)
     - original product_price (原价)
     - flash_price (秒杀价)
     - stock (库存)
     - start_time (开始时间)
     - end_time (结束时间)
     - status (状态: DRAFT, ACTIVE, ENDED, CANCELLED)
     - created_at, updated_at

   - `flash_sale_order`: 秒杀订单表
     - id (PK)
     - activity_id (活动ID)
     - user_id (用户ID)
     - order_no (订单号)
     - status (状态: PENDING, PAID, CANCELLED)
     - created_at, updated_at

2. 创建 JPA 实体类
   - `FlashSaleActivity`
   - `FlashSaleOrder`

3. 创建 Repository 接口
   - `FlashSaleActivityRepository`
   - `FlashSaleOrderRepository`

4. 添加数据库索引
   - flash_sale_activity(product_id)
   - flash_sale_activity(start_time, end_time)
   - flash_sale_order(activity_id, user_id) - 唯一索引

5. 编写数据库单元测试

**依赖**: 无

### 阶段 2：Redis 缓存集成（约 1-2 小时）

**任务列表**:

1. 配置 Redis 连接
   - application.yml 中配置 Redis
   - 创建 RedisTemplate Bean

2. 定义 Redis Key 常量
   - `ACTIVITY_PREFIX = "flash:activity:"`
   - `STOCK_PREFIX = "flash:stock:"`
   - `USER_PREFIX = "flash:user:"`

3. 实现缓存服务
   - `FlashSaleCacheService`
     - `saveActivity()`: 保存活动到 Redis
     - `getActivity()`: 获取活动
     - `initStock()`: 初始化库存
     - `isUserPurchased()`: 检查用户是否已购买

4. 编写 Lua 脚本
   - `decrement_stock.lua`: 原子扣减库存
     - 输入: stock_key, activity_key, user_key, user_id
     - 逻辑: 检查活动状态 → 检查用户是否已购 → 检查库存 → 扣减 → 返回结果

5. 编写缓存集成测试

**依赖**: 阶段 1（数据库设计）

### 阶段 3：核心业务逻辑（约 2-3 小时）

   - `FlashSaleServiceImpl`

       - 如果成功，创建订单
       - 发送通知消息
     - `getActivity()`:
       - 先从 Redis 获取
       - 缓存未命中则查询数据库
     - `cancelActivity()`:
       - 更新活动状态
       - 清理 Redis 缓存

3. 创建 DTO
   - `CreateActivityRequest`
   - `PurchaseRequest`
   - `PurchaseResponse`
   - `ActivityInfo`

4. 编写业务逻辑单元测试
