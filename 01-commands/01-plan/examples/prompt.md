# 用户提示词示例

## 场景描述

在电商平台中实现限时秒杀功能。

## 提示词

```
/plan 为电商平台实现限时秒杀功能，需要支持：

1. 商品秒杀活动的创建、管理和查询
2. 用户在秒杀时间窗口内抢购商品
3. 防止超卖（库存扣减的并发控制）
4. 秒杀结果实时通知
5. 秒杀活动的定时启用和关闭

技术栈：Spring Boot + PostgreSQL + Redis + RabbitMQ
```

## 为什么这个提示词适合 /plan

这个提示词具有以下特点：

### 1. 需求明确但实现复杂

- 功能需求清晰（5个核心功能点）
- 但实现涉及多个技术层面
- 需要多个架构决策

### 2. 涉及技术选择

- 并发控制方案（数据库锁 vs Redis Lua 脚本 vs 分布式锁）
- 缓存策略（如何缓存秒杀活动、如何更新缓存）
- 消息队列用途（订单通知、库存同步）

### 3. 多层架构

- 数据层（秒杀活动表、秒杀订单表）
- 缓存层（Redis 缓存）
- 服务层（秒杀业务逻辑）
- API 层（REST 接口）
- 调度层（定时任务）

### 4. 并发和安全考虑

- 库存扣减的并发控制
- 防止重复下单
- 防止超卖
- 用户限流

### 5. 实现顺序依赖

- 必须先建立数据模型
- 然后实现核心业务逻辑
- 再构建 API 接口
- 最后集成缓存和消息队列

## 可选的提示词变体

### 变体 1：更详细的技术需求

```
/plan 为电商平台实现限时秒杀功能

需求：
- 支持活动创建（商品ID、原价、秒杀价、库存、开始时间、结束时间）
- 用户抢购（单用户限购1件、先到先得、超卖保护）
- 实时库存显示
- 抢购结果通知（成功/失败原因）

技术栈：
- Spring Boot 3.x
- PostgreSQL（主数据存储）
- Redis（缓存 + 分布式锁）
- RabbitMQ（异步通知）

性能要求：
- 支持 1000 QPS 的抢购请求
- 响应时间 < 100ms
- 库存扣减操作原子性保证
```

### 变体 2：强调特定技术方案

```
/plan 使用 Redis + Lua 脚本为电商平台实现秒杀功能

核心功能：
1. 活动管理（CRUD）
2. 库存扣减（Redis Lua 原子操作）
3. 抢购下单
4. 异步通知

技术要求：
- 使用 Redis 实现分布式锁
- 使用 Lua 脚本保证库存扣减原子性
- 使用 RabbitMQ 实现异步通知
- 实现热点数据缓存
```

### 变体 3：重构场景

```
/plan 将现有的秒杀功能从单体架构重构为微服务架构

当前问题：
- 秒杀模块耦合在订单服务中
- 数据库性能瓶颈
- 无法独立扩展秒杀功能

目标架构：
- 独立的秒杀服务
- 使用 Redis 作为主要数据存储
- 通过消息队列与订单服务解耦
```

## 提示词编写建议

### ✅ 好的提示词特征

1. **描述功能，而非实现细节**
   - 好：支持用户抢购商品，防止超卖
   - 不好：使用 Redis 的 decrement 命令扣减库存

2. **包含业务约束**
   - 好：单用户限购1件，支持1000 QPS
   - 不好：实现秒杀功能

3. **提供技术栈上下文**
   - 好：使用 Spring Boot + PostgreSQL + Redis
   - 不好：不提供技术栈信息

4. **明确非功能性需求**
   - 好：响应时间 < 100ms，保证库存扣减原子性
   - 不好：要快一点

### ❌ 避免的提示词特征

1. **过于简单**
   - 不好：实现秒杀功能
   - 原因：缺少上下文，规划代理无法生成详细计划

2. **指定实现细节**
   - 不好：使用 Redis 的 INCRBY 命令实现库存
   - 原因：限制了规划代理的技术决策空间

3. **缺少技术栈信息**
   - 不好：添加一个定时任务功能
   - 原因：无法确定使用什么工具（Spring Scheduler、Quartz 等）

4. **一次性写太多需求**
   - 不好：在一个提示词中包含10个以上功能
   - 原因：计划会过于复杂，难以管理

## 练习任务

尝试自己编写一个 `/plan` 提示词，要求：

1. 选择一个你熟悉的业务场景
2. 包含至少3个核心功能
3. 说明技术栈和关键约束
4. 确保提示词清晰且不指定实现细节

然后执行 `/plan` 命令，观察生成的计划是否符合预期。
