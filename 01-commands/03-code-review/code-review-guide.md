# Code Review 指南

代码审查是确保代码质量、分享知识和维护代码库健康的重要实践。

## 📌 什么是 Code Review

Code Review 是在代码合并到主分支之前，由其他开发者检查代码的过程。它的目标包括：

- **发现缺陷** - 在代码运行前发现潜在问题
- **确保一致性** - 维持代码风格和架构一致性
- **知识分享** - 团队成员互相学习最佳实践
- **提高可维护性** - 确保代码易于理解和修改
- **建立团队标准** - 通过审查强化编码标准

### 审查范围

Code Review 应该关注：

- **安全性** - SQL 注入、XSS、认证/授权问题
- **正确性** - 逻辑错误、边界条件、异常处理
- **可读性** - 命名、注释、代码组织
- **性能** - 算法复杂度、资源使用
- **测试** - 测试覆盖、测试质量
- **一致性** - 符合项目规范

## 🎯 审查最佳实践

### 作为审查者

1. **及时响应** - 不要让 PR 等待
2. **具体反馈** - 提供"为什么"和"如何改进"
3. **区分优先级** - CRITICAL > HIGH > MEDIUM > LOW
4. **关注重要问题** - 不要纠结个人偏好
5. **建议而非命令** - 使用建议性语气

### 作为被审查者

1. **自检提交** - 提交前自己审查
2. **保持小 PR** - 一次改动不要太大
3. **清晰描述** - 解释改动的原因
4. **回应反馈** - 及时回应评论
5. **学习改进** - 从反馈中学习

## 🔍 审查检查清单

### 通用检查

- [ ] 代码符合项目编码规范
- [ ] 变量和函数命名清晰
- [ ] 没有硬编码的敏感信息
- [ ] 错误处理完善
- [ ] 没有调试代码（console.log, debugger）
- [ ] 注释准确且必要
- [ ] 没有重复代码
- [ ] 复杂度在合理范围

### 安全检查（CRITICAL）

- [ ] 没有 SQL 注入风险
- [ ] 没有 XSS 漏洞
- [ ] 认证和授权正确
- [ ] 敏感数据不泄露到日志
- [ ] 密码正确哈希
- [ ] 没有 CSRF 漏洞

### 功能检查（HIGH）

- [ ] 业务逻辑正确
- [ ] 边界条件处理
- [ ] 输入验证完整
- [ ] 异常处理覆盖
- [ ] 没有竞态条件

### 质量检查（MEDIUM）

- [ ] 代码可读性好
- [ ] 函数职责单一
- [ ] 没有魔法数字
- [ ] 使用合适的数据结构
- [ ] 性能合理

### 优化检查（LOW）

- [ ] 可以优化的算法
- [ ] 可以重构的结构
- [ ] 可以改进的命名

## 📝 Code Review 模板

### PR 描述模板

```markdown
## 变更类型
- [ ] 新功能
- [ ] Bug 修复
- [ ] 重构
- [ ] 文档更新
- [ ] 性能优化

## 描述
简要描述这次变更的目的和内容

## 相关 Issue
Closes #123, #456

## 测试计划
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 手动测试完成
- [ ] 性能测试（如适用）

## 截图/录屏
（如果是 UI 变更，提供截图或录屏）

## 审查要点
请特别关注以下方面：
1. [具体关注点 1]
2. [具体关注点 2]
```

### Review 反馈模板

```markdown
### 问题类型
- [ ] CRITICAL - 必须修复
- [ ] HIGH - 强烈建议修复
- [ ] MEDIUM - 建议修复
- [ ] LOW - 可选优化
- [ ] INFO - 信息性评论

### 问题描述
[具体问题描述]

### 位置
文件:filename.js:行号

### 建议
[具体的改进建议]

### 原因
[为什么这样建议]

### 替代方案
[如果有的话]
```

## 🎓 审查技巧

### 1. 使用 IDE 工具

- GitHub PR Review 辅助
- GitLens (VS Code)
- SonarLint

### 2. 分层次审查

**第一遍：快速浏览**
- 整体结构
- 命名风格
- 明显问题

**第二遍：深入理解**
- 业务逻辑
- 边界条件
- 错误处理

**第三遍：细节检查**
- 性能
- 安全
- 最佳实践

### 3. 关注原则

**关注接口而非实现**
- 公共 API 的设计最重要
- 内部实现可以灵活

**关注可维护性**
- 未来维护者能理解吗
- 容易扩展吗

**关注测试**
- 测试覆盖重要路径
- 测试质量如何

## 📊 Code Review 指标

### 质量指标

- **问题密度** - 每千行代码的问题数
- **修复时间** - 问题发现到修复的时间
- **审查时间** - PR 创建到首次审查的时间
- **通过率** - 一次通过审查的 PR 比例

### 团队健康指标

- **参与度** - 团队成员参与审查的比例
- **审查深度** - 评论数量和详细程度
- **响应速度** - 平均审查响应时间

## 🚫 常见误区

### 作为审查者

❌ **个人偏好** - "我喜欢用 const，不用 let"
✅ **代码规范** - "项目规范要求使用 const"

❌ **完美主义** - 要求所有问题都修复
✅ **优先级排序** - 先修 CRITICAL，再修 HIGH

❌ **模棱两可** - "这段代码不太对"
✅ **具体明确** - "在文件名.js:行号，应该使用参数化查询"

### 作为被审查者

❌ **防御性** - 解释所有问题而不修复
✅ **建设性** - 理解反馈并改进

❌ **提交前不测试** - 相信 CI 会发现问题
✅ **自测提交** - 提交前自己运行测试
