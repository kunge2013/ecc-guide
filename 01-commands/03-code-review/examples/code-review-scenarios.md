# Code Review 场景分析

## Code Review 的核心价值

### 1. 质量保证
- 在代码运行前发现缺陷
- 确保代码符合项目标准
- 防止技术债积累

### 2. 知识共享
- 团队成员互相学习
- 最佳实践传播
- 隐式知识显性化

### 3. 架构一致性
- 维持设计模式一致性
- 防止架构腐化
- 确保接口契约稳定

### 4. 安全防护
- 发现安全漏洞
- 验证权限控制
- 检查敏感数据处理

## 什么时候进行 Code Review

### 必须进行（CRITICAL）

1. **安全相关代码**
   - 认证/授权
   - - �数据处理
   - 输入验证
   - 加密/解密

2. **核心业务逻辑**
   - 支付处理
   - 账务计算
   - 用户状态变更

3. **公共 API 变更**
   - API 接口修改
   - 数据结构变更
   - 协议变更

4. **性能关键代码**
   - 数据库查询
   - 算法实现
   - 缓存策略

5. **架构级变更**
   - 设计模式引入
   - 模块重构
   - 依赖注入变更

### 建议进行（HIGH）

1. **复杂的逻辑实现**
   - 状态机
   - 并发控制
   - 异步处理

2. **配置和环境变更**
   - 环境变量修改
   - 配置文件更新
   - 依赖升级

3. **用户体验代码**
   - UI 组件
   - 用户流程
   - 错误消息

### 可选进行（LOW）

1. **文档更新**
   - README 修改
   - 注释更新

2. **测试代码**
   - 单元测试添加
   - 测试工具脚本

3. **重构（已测试）**
   - 简单重命名
   - 格式调整

## Code Review 的优先级

| 优先级 | 类型 | 处理时间 | 示例 |
|--------|------|----------|------|
| 🔴 CRITICAL | 阻止合并 | 立即 | 安全漏洞、数据损坏风险 |
| 🟠 HIGH | 强烈建议 | 24小时内 | 性能问题、架构不一致 |
| 🟡 MEDIUM | 建议 | 3天内 | 代码风格、命名改进 |
| 🟢 LOW | 可选 | 1周内 | 注释完善、微优化 |

## Code Review 的常见类型

### 1. 实施前审查（Implementation Review）

**时机：** 代码编写之前

**内容：**
- 设计评审
- 架构决策
- API 契约定义

**工具：**
- ADR (Architecture Decision Records)
- 设计文档
- 原型评审

### 2. 代码变更审查（Change Review）

**时机：** PR 创建后、合并前

**内容：**
- 代码质量
- 安全性
- 功能正确性
- 测试覆盖

**工具：**
- GitHub PR Review
- GitLab Merge Request
- Phabricator

### 3. 架构审查（Architecture Review）

**时机：** 重大架构变更

**内容：**
- 设计模式
- 模块划分
- 依赖关系
- 可扩展性

**工具：**
- 架构评审会议
- 技术委员会审查
- 设计文档评审

### 4. 安全审查（Security Review）

**时机：** 安全相关代码变更

**内容：**
- 漏洞扫描
- 依赖审计
- 渗透测试

**工具：**
- Snyk
- OWASP ZAP
- SonarQube

## 不同场景的 Code Review 策略

### 1. 紧急 Bug 修复

**特点：**
- 时间压力大
- 范围应该很小
- 需要快速审查

**策略：**
- ✅ 限制变更范围（<50 行）
- ✅ 至少一人审查
- ✅ 关注安全和回归
- ✅ 后续补充测试

**不建议：**
- ❌ 跳过审查直接合并
- ❌ 在紧急修复中引入新功能
- ❌ 不写测试就合并

### 2. 新功能开发

**特点：**
- 实现复杂度可变
- 需要深入理解
- 影响范围大

**策略：**
- ✅ 小 PR（<400 行变更）
- ✅ 至少两人审查
- ✅ 包含测试
- ✅ 更新文档

**检查重点：**
- API 设计
- 错误处理
- 边界条件
- 性能影响

### 3. 重构

**特点：**
- 行为应该保持不变
- 可能涉及多个文件
- 测试是关键

**策略：**
- ✅ 确保测试覆盖
- ✅ 分步重构
- ✅ 使用自动化工具
- ✅ 对比性能

**检查重点：**
- 行为不变性
- 测试质量
- 性能对比
- 可读性提升

### 4. 第三方库升级

**特点：**
- 外部依赖变更
- 可能引入破坏性变更
- 安全补丁优先级高

**策略：**
- ✅ 查看变更日志
- ✅ 关注破坏性变更
- ✅ 运行完整测试套件
- ✅ 安全升级优先

**检查重点：**
- 破坏性变更
- 安全漏洞修复
- 性能变化
- 兼容性

## Code Review 的效率优化

### 1. 提高审查者效率

**技术层面：**
- 使用代码差异工具
- 配置 IDE 插件
- 使用代码质量工具自动检查

**流程层面：**
- 建立检查清单
- 明确审查范围
- 优先处理安全关键代码

**沟通层面：**
- 使用模板化反馈
- 提供上下文信息
- 给出可操作的建议

### 2. 降低审查者负担

**作者责任：**
- 保持 PR 小而聚焦
- 写清晰的描述
- 运行所有本地测试
- 自我审查

**自动化工具：**
- Linting 自动修复
- 格式化工具
- 静态分析
- 测试覆盖率检查

### 3. 平衡速度和质量

**快速通道：**
- 小改动 (<50 行)
- 纯文档更新
- 测试代码补充

**标准通道：**
- 常规功能开发
- Bug 修复
- 小型重构

**慢速通道：**
- 架构变更
- 大型重构
- 新模块引入
