# planner 代理

## 概述

planner 代理是 ECC 的核心代理之一，负责在编写代码之前创建详细的实现计划。当用户执行 `/plan` 命令时，planner 代理会被自动调用。

## 代理职责

planner 代理的主要职责包括：

1. **分析需求** - 理解用户的需求和目标
2. **重述需求** - 以清晰、明确的方式重述需求
3. **技术方案设计** - 提出技术实现方案
4. **分解实现步骤** - 将复杂任务分解为可管理的步骤
5. **识别依赖关系** - 识别任务之间的依赖
6. **评估风险** - 识别潜在风险和问题
7. **创建计划文档** - 生成结构化的计划文档
8. **等待用户确认** - 等待用户确认或修改计划

## 触发条件

planner 代理在以下情况下被触发：

1. **用户执行 `/plan` 命令** - 最常见的触发方式
2. **其他代理调用** - 某些复杂场景下，其他代理可能调用 planner
3. **工作流自动触发** - 在特定工作流中自动规划

## 工作流程

### 标准规划流程

```
1. 需求分析
   ↓
2. 需求重述
   ↓
3. 技术方案设计
   ↓
4. 实现步骤分解
   ↓
5. 依赖关系识别
   ↓
6. 风险评估
   ↓
7. 计划文档生成
   ↓
8. 用户确认
   ↓
9. 计划执行（如果确认）
```

### 详细步骤

#### 步骤 1: 需求分析

planner 代理首先分析用户的需求：

- 理解功能目标
- 识别关键要求
- 确定约束条件
- 了解现有项目结构

**输入示例**:
```
用户: /plan 添加用户标签功能，支持多标签和标签筛选
```

**分析结果**:
- 功能：用户标签管理
- 关键要求：多标签支持、标签筛选
- 约束：与现有任务系统集成
- 项目结构：Spring Boot MVC 架构

#### 步骤 2: 需求重述

以清晰、结构化的方式重述需求：

**示例**:
```markdown
## 需求重述

为任务管理系统添加用户标签功能：

### 功能要求
- 用户可以创建、编辑、删除标签
- 标签具有名称和颜色属性
- 一个任务可以关联多个标签
- 可以按标签筛选任务列表

### 成功标准
- 用户能够管理标签
- 任务能够正确关联标签
- 标签筛选功能正常工作
- 所有功能都有完整的测试
```

#### 步骤 3: 技术方案设计

提出技术实现方案：

**示例**:
```markdown
## 技术方案

### 数据模型
- 创建 Tag 实体（id, name, color, userId, createdAt）
- 创建 Task 和 Tag 的多对多关系（TaskTag 实体）

### API 设计
- POST /api/tags - 创建标签
- PUT /api/tags/{id} - 更新标签
- DELETE /api/tags/{id} - 删除标签
- GET /api/tasks?tagId={id} - 按标签筛选任务
- POST /api/tasks/{taskId}/tags - 为任务添加标签
- DELETE /api/tasks/{taskId}/tags/{tagId} - 移除任务标签
```

#### 步骤 4: 实现步骤分解

将实现分解为可管理的阶段：

**示例**:
```markdown
## 实现步骤

### 阶段 1：数据库模型
1. 创建 Tag 实体
2. 创建 TaskTag 关联实体
3. 创建 TagRepository
4. 更新 Task 实体（添加标签关联）
5. 创建数据库迁移脚本

### 阶段 2：服务层
1. 创建 TagService
2. 实现标签 CRUD 操作
3. 实现标签验证逻辑
4. 更新 TaskService 支持标签操作

### 阶段 3：API 层
1. 创建 TagController
2. 实现标签 API 端点
3. 更新 TaskController 添加标签操作
4. 添加输入验证和错误处理

### 阶段 4：测试
1. 编写单元测试（Service 层）
2. 编写集成测试（API 层）
3. 验证标签功能完整性
```

#### 步骤 5: 依赖关系识别

识别任务之间的依赖：

**示例**:
```markdown
## 依赖关系

### 内部依赖
- Tag 实体创建 → TaskTag 关联实体
- TagService 创建 → TagController
- 数据库迁移 → 实体测试

### 外部依赖
- 颜色验证可以使用第三方库（如 Color.js）
- 无外部服务依赖

### 前置条件
- 项目已有 Task 实体
- 数据库连接已配置
```

#### 步骤 6: 风险评估

识别潜在风险和缓解措施：

**示例**:
```markdown
## 风险分析

### 技术风险
- **风险**: 多对多关系处理复杂
  - **影响**: 中等
  - **缓解**: 使用 JPA 的 @ManyToMany 注解

- **风险**: 颜色格式验证
  - **影响**: 低
  - **缓解**: 使用正则表达式验证 HEX 颜色

### 业务风险
- **风险**: 标签名称冲突
  - **影响**: 中等
  - **缓解**: 添加标签名称唯一性约束

### 实现风险
- **风险**: 标签删除时的关联处理
  - **影响**: 高
  - **缓解**: 实现级联删除或软删除
```

#### 步骤 7: 计划文档生成

生成结构化的计划文档：

**示例**:
```markdown
# 实现计划：用户标签功能

## 需求重述
[步骤 2 的内容]

## 技术方案
[步骤 3 的内容]

## 实现步骤
[步骤 4 的内容]

## 依赖关系
[步骤 5 的内容]

## 风险分析
[步骤 6 的内容]

## 预估复杂度
- 复杂度：中等
- 预估时间：4-6 小时
- 测试时间：2-3 小时
```

#### 步骤 8: 用户确认

等待用户确认或修改计划：

**交互示例**:
```
ECC: [展示计划文档]

ECC: 等待确认：proceed? (yes/no/modify)
```

## 代理特性

### 智能规划

planner 代理会考虑：

- **项目上下文** - 根据项目结构和现有代码调整计划
- **技术栈** - 根据使用的技术（如 Spring Boot、Django）调整方案
- **最佳实践** - 遵循行业最佳实践和编码规范
- **可维护性** - 确保计划考虑代码的可维护性

### 灵活调整

planner 代理支持：

- **需求修改** - 用户可以要求修改计划
- **步骤调整** - 可以添加、删除、重新排序步骤
- **详细程度调整** - 可以增加或减少计划的详细程度

### 风险意识

planner 代理会：

- **识别常见风险** - 基于经验识别常见问题
- **提供缓解措施** - 为每个风险提供具体的缓解方案
- **评估风险等级** - 评估风险的严重程度和影响

## 与其他代理的协作

planner 代理会与其他代理协作：

### 与 tdd-guide 协作

```
planner 生成计划
    ↓
用户确认计划
    ↓
tdd-guide 指导 TDD 实现
```

### 与 code-reviewer 协作

```
planner 生成计划
    ↓
用户实现代码
    ↓
code-reviewer 审查代码（参考计划）
```

## 最佳实践

### 对于用户

1. **清晰描述需求** - 详细描述你想要实现的功能
2. **提供上下文** - 说明相关的项目背景和约束
3. **审查计划** - 仔细检查生成的计划
4. **主动修改** - 不要害怕要求修改计划
5. **遵循计划** - 计划经过深思熟虑，建议遵循

### 对于理解 planner

1. **它是自动化规划工具** - 不是魔法，而是基于最佳实践
2. **它考虑项目上下文** - 计划会根据项目调整
3. **它支持迭代** - 可以多次修改和完善计划
4. **它不是一成不变的** - 计划可以在实现过程中调整
