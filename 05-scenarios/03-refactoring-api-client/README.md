# 重构 API 客户端

这个场景展示了如何使用 Claude Code 重构 API 客户端，改进架构和可维护性。

## 🎯 场景概述

### 适用情况

- API 客户端代码混乱
- 缺乏类型安全
- 错误处理不一致
- 难以测试和维护

### 核心挑战

- 保持向后兼容性
- 改进类型安全
- 统一错误处理
- 提高可测试性

## 📋 使用场景

### 场景 1: 重构为 TypeScript

**问题：** JavaScript API 客户端缺乏类型安全

**改进：**
- 添加 TypeScript 类型定义
- 使用泛型
- 改进类型推断

**Agent 使用：**
- Planner Agent 设计类型系统
- TDD Guide Agent 确保重构正确
- Code Reviewer Agent 验证类型安全

---

### 场景 2: 统一错误处理

**问题：** 各个 API 方法错误处理不一致

**改进：**
- 创建统一错误类
- 实现重试机制
- 添加日志记录

**Agent 使用：**
- 分析现有错误处理
- 设计错误处理架构
- 实现重构

---

### 场景 3: 添加缓存层

**问题：** 频繁的 API 调用影响性能

**改进：**
- 实现缓存策略
- 添加缓存失效机制
- 集成 Redis 或内存缓存

**Agent 使用：**
- 设计缓存架构
- 实现缓存层
- 性能测试验证

## 🚀 重构流程

### 步骤 1: 分析和规划

```bash
Agent(planner, """
分析现有 API 客户端代码，制定重构计划：

当前问题：
- 缺乏类型安全
- 错误处理不一致
- 没有缓存机制
- 难以测试

请制定重构计划，包括：
1. 类型系统设计
2. 错误处理架构
3. 缓存策略
4. 向后兼容性考虑
""")
```

### 步骤 2: 渐进式重构

```bash
Agent(tdd-guide, """
分阶段重构 API 客户端：

阶段 1: 添加 TypeScript 类型定义
阶段 2: 统一错误处理
阶段 3: 添加缓存层
阶段 4: 改进测试覆盖

每个阶段使用 TDD：
1. 先写测试
2. 实现代码
3. 验证功能

确保重构过程中保持向后兼容性。
""")
```

### 步骤 3: 验证和测试

```bash
Agent(e2eater, """
测试重构后的 API 客户端：

1. 功能测试
   - 所有 API 端点正常工作
   - 错误处理正确
   - 缓存按预期工作

2. 性能测试
   - 缓存命中率
   - 响应时间
   - 并发处理

3. 兼容性测试
   - 现有代码仍能工作
   - API 契约不变
   - 错误行为一致
""")
```

## 💡 重构原则

### 1. 小步前进

- 每次只重构一个小部分
- 每步都运行测试验证
- 保持代码始终可运行

### 2. 保持兼容性

- 不破坏现有 API
- 使用废弃标记移除旧功能
- 提供迁移指南

### 3. 测试驱动

- 先写测试验证当前行为
- 重构代码通过测试
- 确保行为不变

### 4. 持续集成

- 每次 PR 运行完整测试
- 自动化代码质量检查
- 监控生产性能
