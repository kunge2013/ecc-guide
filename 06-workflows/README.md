# 工作流程指南

## 概述

本目录包含 ECC 在不同场景下的工作流程指南，帮助你快速选择合适的工具和方法。

## 工作流程分类

### 1. 功能开发流程
**文件**: [feature-development.md](./feature-development.md)

**适用场景**:
- 开发新功能
- 添加新 API 端点
- 实现新业务逻辑

**主要步骤**:
```
1. /plan          → 制定实现计划
2. /tdd           → 测试驱动实现
3. /code-review   → 代码审查
4. /e2e           → 端到端测试
5. git commit     → 提交代码
```

### 2. Bug 修复流程
**文件**: [bug-fix.md](./bug-fix.md)

**适用场景**:
- 修复已知的 Bug
- 处理生产环境问题
- 解决用户反馈

**主要步骤**:
```
1. 重现问题
2. /plan          → 制定修复计划
3. /tdd           → 测试驱动修复
4. /code-review   → 代码审查
5. 验证修复
6. git commit     → 提交修复
```

### 3. 重构流程
**文件**: [refactoring.md](./refactoring.md)

**适用场景**:
- 代码重构
- 改进代码结构
- 提取公共逻辑
- 优化性能

**主要步骤**:
```
1. 分析现有代码
2. /plan          → 制定重构计划
3. 编写/更新测试
4. 执行重构
5. /code-review   → 代码审查
6. 运行测试验证
7. git commit     → 提交重构
```

### 4. 决策树
**文件**: [decision-tree.md](./decision-tree.md)

**用途**: 根据任务类型快速选择合适的 ECC 工具

**包含内容**:
- 快速决策树
- 任务类型分类
- 工具选择建议
- 使用示例

### 5. 最佳实践
**文件**: [best-practices.md](./best-practices.md)

**包含内容**:
- ECC 使用最佳实践
- 常见错误和避免方法

- 性能优化技巧

## 完整工作流程示例

### 场景：开发新的 REST API 端点

```bash
# 1. 规划阶段
用户: /plan 我需要添加一个 REST API 端点，用于获取用户的任务统计信息

ECC: [planner 代理]
- 分析需求
- 识别依赖
- 制定实现计划
- 等待确认

用户: yes  # 确认计划

# 2. 实现阶段
用户: /tdd 实现任务统计 API 端点

ECC: [tdd-guide 代理]
- 指导编写测试（RED）
- 指导实现功能（GREEN）
- 指导重构优化（IMPROVE）

# 3. 审查阶段
用户: /code-review 审查任务统计 API 的代码

ECC: [code-reviewer 代理]
- 检查代码质量
- 识别安全问题
- 验证最佳实践
- 提供改进建议

# 4. 测试阶段
用户: /e2e 生成并运行 API 端点测试

ECC: [e2e-runner 代理]
- 生成测试脚本
- 运行端到端测试
- 验证功能

# 5. 提交阶段
用户: git add .
用户: git commit -m "feat: 添加任务统计 API 端点"
```

## 工作流程最佳实践

### 1. 总是从 `/plan` 开始

**原因**:
- 明确需求和实现步骤
- 识别潜在风险
- 提高开发效率

**示例**:
```bash
# 好的做法
用户: /plan 添加用户认证功能
用户: /tdd 实现用户认证功能

# 不好的做法
用户: 直接开始写代码，没有规划
```

### 2. 遵循 TDD 原则

**原因**:
- 保证代码质量
- 避免遗漏测试
- 降低返工成本

**示例**:
```bash
# 好的做法
用户: /tdd 实现功能

# 不好的做法
用户: 先写实现，再写测试
```

### 3. 不要跳过代码审查

**原因**:
- 发现潜在问题
- 保证代码质量
- 学习最佳实践

**示例**:
```bash
# 好的做法
用户: /code-review 审查代码

# 不好的做法
用户: 直接提交代码，跳过审查
```

### 4. 完成功能后使用 `/learn`

**原因**:
- 提取可重用模式
- 积累知识库
- 提高团队效率

**示例**:
```bash
# 好的做法
用户: /learn 提取用户认证模式

# 不好的做法
用户: 完成功能后不做知识提取
```

### 团队成员分工

| 角色 | 主要职责 | 常用工具 |
|-----|---------|---------|
| **开发者** | 功能开发、Bug 修复 | `/plan`, `/tdd` |

| **QA 工程师** | 测试验证 | `/e2e` |
| **技术负责人** | 架构设计、知识管理 | `/plan`, `/learn` |

### 团队协作流程

```
1. 技术负责人: /plan 制定实现计划
2. 开发者: /tdd 实现功能
3. 开发者: /code-review 自我审查
4. 代码审查者: /code-review 团队审查
5. QA 工程师: /e2e 测试验证
6. 技术负责人: /learn 提取可重用模式
```
